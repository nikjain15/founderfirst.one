# Screen brief 08 — Error, empty, loading, offline states

*Every screen owes the user a graceful answer when things go wrong.*

This brief covers the cross-cutting states the demo must handle.
Individual screens can override copy, but the **shapes and the rules**
defined here are the floor.

---

## 1. Loading states

Loading happens for two reasons in the demo:

- A Penny utterance is being generated (`renderPenny` call in flight).
- A prompt or config file is being fetched.

### Rule: never show a spinner next to Penny.

Spinners look like the app is broken. Use a **typing dots** element
inside a Penny bubble:

```
<Bubble from="penny">
  <TypingDots />
</Bubble>
```

### Rule: loading state must be visible within 150 ms.

If the response comes back sooner than that, skip it — a flash of
loading is worse than no loading.

### Rule: loading state has no copy.

No "Penny is thinking…", no "One moment". The dots speak for themselves.

---

## 2. Empty states

Empty states happen when:

- The user reaches My Books with no transactions yet.
- The user reaches Needs a look with nothing needing review.
- The user opens a drill-down (invoices, expenses) with no records.

### Rule: empty is never shame.

**Banned:** "You have no transactions yet."
**Banned:** "Nothing to do."
**Banned:** "Get started by…"
**OK:** "I haven't seen anything come through yet. I'll be here when it does."

Copy is generated by Penny via `renderPenny({ intent: "thread.idle" })`
with an `empty: true` flag in context. Empty state copy is not
hard-coded in the screen.

### Rule: empty states always offer one concrete next step.

One, not three. Examples:

- My Books (no transactions): "Connect a bank to get started" → deep-link
  to Add tab, Connect section.
- Needs a look (nothing to review): "You're clear. I'll ping you when
  something lands." No CTA. This is a success state.
- Invoices drill-down (no invoices): "Want to send your first invoice?"
  → deep-link to Add tab, Invoice.

---

## 3. Error states

Error states happen when:

- `renderPenny` fails after all retries (validator kept rejecting).
- The Worker returns a non-200 (rate limited, upstream outage).
- A prompt file 404s (bug — should not happen in prod).
- A config file is malformed (bug — should not happen in prod).

### Rule: never blame Claude, Anthropic, the Worker, or "the AI".

**Banned:** "Claude couldn't respond."
**Banned:** "AI error."
**Banned:** "Our AI had an issue."
**OK:** "Something's off on my end. Give me a second and try again."

### Rule: never show a stack trace, a request id, or an HTTP status code
to the user. Log it to the console for the developer; show the user a
calm single line.

### Rule: errors always show a retry button.

- For `renderPenny` failures: "Try again" re-calls with `nocache: true`.
- For Worker 429 (rate limit): "I'm catching up. Try again in a few
  seconds." No button — auto-retry after backoff.
- For Worker 5xx: "Try again" with retry.

### Rule: errors never dismiss user input.

If the user typed something into a capture field and the parse failed,
their text stays. They should not have to re-type.

---

## 4. Offline states

The demo can be opened on a flaky connection (airport wifi, subway).

### Rule: detect offline via `navigator.onLine` + a ping check to the
Worker on boot. Cache the last-known state in `penny-demo-state-v5`
(already in place via App.jsx).

### Rule: when offline, show a single persistent banner above the tab
bar:

```
You're offline. Penny will catch up when you're back.
```

### Rule: offline never silently drops user actions.

If the user taps "Confirm" on an approval card while offline, queue
the action locally and replay it when connectivity returns. Show a
small pending dot on the card until it replays.

### Rule: offline never blocks navigation.

The user can still move between Penny, Add, and My Books and see
cached data. Only live AI calls fail.

---

## 5. First-run empty state (before bank connect)

After onboarding finishes but before the demo "connects" the fake bank,
the Penny tab shows a greeting plus a pending expectation — "I'll start
seeing your money move in a second." This is a scripted first-run
moment, not a generic empty state, and it should feel warm.

Transition: after 3 seconds, the scenario engine starts dropping events
and the first approval card appears.

---

## 6. What components need to exist

Build these in `components/`:

| Component | Purpose |
|---|---|
| `<TypingDots />` | Three-dot loading indicator inside a Penny bubble. |
| `<EmptyState />` | Empty-state layout: icon-free, single line of copy, one CTA. |
| `<ErrorState />` | Calm error line + retry button. Takes a `onRetry` prop. |
| `<OfflineBanner />` | Persistent top banner. Auto-hides when back online. |

---

## 7. What the Claude Code session builds

1. Build the four components above in `components/`.
2. Wire them into every screen that makes an AI call or shows a list.
3. Add `tests/empty-states.test.js` — one case per screen asserting
   empty copy is Penny-generated, not hard-coded.
