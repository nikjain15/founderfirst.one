# Email registry — every email FounderFirst sends

The single source of truth for *what* we email, *why*, *who generates it*, and *when*.
**Rule: no new email ships without a row here.** When you add or change an email,
update this file in the same commit.

Last verified: 2026-06-29 (added admin welcome; registry caught up with welcome + custom/scheduled emails).

## Live emails

### 1. Signals daily digest
- **Purpose:** Surface the day's new high-intent leads + competitor mentions so the team can reach out fast.
- **Generated by:** `supabase/functions/listening-digest/` → Resend
- **Trigger:** pg_cron job `signals-daily-digest` (`20260622110000_signals_digest.sql`) — fires daily; the function decides whether to send.
- **Cadence:** Cron fires daily **13:00 UTC**, but the email only sends when a lead clears the intent threshold (`SIGNALS_DIGEST_INTENT_MIN`, default 70) **or** the 7-day floor is hit. Lookback window anchors to the last send (`sig_digest_sends`, `20260623150000_signals_digest_sends.sql`) so quiet days stay silent without dropping leads. Sends nothing if nothing accumulated.
- **Audience:** All admins (`admins` table).
- **Subject:** `N new leads · top intent T` (number-led, hottest score)
- **Eyebrow:** FounderFirst · Signals
- **CTA → :** `/admin/audience#signals` ("Open Signals")

### 2. What's new — weekly changelog
- **Purpose:** Tell the team what shipped this week. Review-then-send: the cron only *nudges*; an admin clicks to actually send.
- **Generated by:** `supabase/functions/changelog-digest/` → Resend
- **Trigger:** pg_cron `changelog-weekly-digest` (`20260623120000_changelog.sql`) for the nudge; admin click in **How it works → What's new** for the real send.
- **Cadence:** Nudge every **Monday 13:00 UTC**; digest send is manual. Nothing sent if nothing shipped.
- **Audience:** All admins.
- **Subjects:** nudge `N updates ready to send` · digest `N things shipped this week` (number-led)
- **Eyebrow:** FounderFirst · What's new
- **CTA → :** `/admin/how-it-works#whats-new` (digest "See what shipped" · nudge "Review & send")

### 3. Penny's brain updated
- **Purpose:** Alert admins when someone publishes a new Voice or System-prompt version (it goes live on every Penny surface immediately).
- **Generated by:** `supabase/functions/notify-content-change/` → Resend
- **Trigger:** DB `after_publish_*` triggers via pg_net (`20260621100000_publish_notify.sql`).
- **Cadence:** Event-driven, on each publish.
- **Audience:** All admins **except** the person who published.
- **Subject:** `<Voice guide|System prompt> vN is live` (eyebrow carries the "Penny's brain" context)
- **Eyebrow:** Penny's brain
- **CTA → :** `/admin/content/#voice` or `#prompt` ("Review the change")

### 4. Admin welcome
- **Purpose:** Tell a newly-added admin they have access and how to sign in (one-tap magic link). Sent once per address.
- **Generated by:** `supabase/functions/admin-welcome/` → Resend (shared send path).
- **Trigger:** The admin app's "Invite admin →" calls it right after the `admins` insert (`inviteAdmin` in `apps/admin/src/lib/supabase.ts`). Best-effort; a send failure never fails the add.
- **Cadence:** Event-driven, on each add. Idempotent via `admin_welcome_sends` ledger (re-add never double-sends).
- **Audience:** The one new admin.
- **Subject:** `You've got FounderFirst admin access`
- **Eyebrow:** FounderFirst · Admin
- **CTA → :** `/admin` ("Open the admin"). Toggle: Settings → Emails → Scheduled (`admin_welcome`).

### 5. Waitlist welcome
- **Purpose:** Confirm a new waitlist signup and set expectations. Sent once per address.
- **Generated by:** `supabase/functions/signup-confirmation/` → Resend (shared send path).
- **Trigger:** Public signup island calls it after `signup_to_waitlist`. Idempotent via `welcome_sends` ledger.
- **Cadence:** Event-driven, on each new signup.
- **Audience:** The new signup.
- **Subject:** `You're on the list, {firstName}`
- **Eyebrow:** FounderFirst
- **CTA → :** `/confirmed` ("See your spot"). Toggle: Settings → Emails → Scheduled (`welcome`).

### 6. Custom + scheduled emails  *(admin-authored)*
- **Purpose:** One-off or recurring emails an admin composes in Settings → Emails (Templates → Scheduled).
- **Generated by:** `supabase/functions/email-dispatch/` (hourly cron; `dispatch='generic'` renders+sends, `dispatch='invoke'` calls a built-in's specialised fn) + `email-test/` for single test sends. All via the shared send path.
- **Cadence:** Per the `email_schedules` row (the single source of truth for timing).
- **Audience:** Per the schedule's audience.
- **Note:** Copy + brand are fully admin-editable; the shell/deliverability stay in code.

### 7. Magic-link sign-in  *(externally managed — already on-brand)*
- **Purpose:** Passwordless admin login — "one email, one link."
- **Generated by:** **Supabase Auth** (not our code), so it does **not** go through `emailShell()`. The on-brand source of truth is [`auth-magic-link.html`](auth-magic-link.html).
- **Trigger:** Admin requests a sign-in link. **Cadence:** On demand. **Audience:** Whoever is signing in.
- **Status:** **Deployed on-brand.** The live Auth template mirrors `auth-magic-link.html` (subject "Your FounderFirst sign-in link"). To re-deploy a change, PATCH `https://api.supabase.com/v1/projects/{ref}/config/auth` with `mailer_templates_magic_link_content` (+ `mailer_subjects_magic_link`) using an `sbp_` Management-API token — no dashboard paste needed.

## Shared infrastructure

- **Render layer:** `_shared/email.ts` (`emailShell()` + `BRAND`). All app-generated emails (1–3) use it.
- **Standard / rulebook:** [`EMAIL.md`](EMAIL.md).
- **Send layer:** Resend API, `from = NOTIFY_FROM` secret (verified `founderfirst.one` sender).
- **Recipients source:** the `admins` table.

## How to read a trigger
- **pg_cron** = a scheduled job inside Postgres. Find the schedule string (e.g. `0 13 * * 1` = Mondays 13:00 UTC) in the named migration.
- **DB trigger** = fires on a row change (e.g. a publish), not a clock.
- **Externally managed** = generated by a third party (Supabase Auth), configured outside this repo.
