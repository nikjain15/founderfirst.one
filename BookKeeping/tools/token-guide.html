Token Cost Guide — Penny Project

How Claude consumes tokens, what drives cost and quality, and how to work efficiently.

⚡
The one thing to understand
Claude pays for everything in the conversation window — not just what you type. Every message I've sent, every file I've read, every document in context: they all count as input tokens on every single turn. A 200k context window sounds huge, but a session reading several large project docs + long back-and-forth can fill it in 30–40 exchanges. When the window fills, old messages drop off — and I lose context of earlier decisions.

Note: CLAUDE.md helps because I read it once and don't need to re-read other files to recover context. Without it, I'd re-read everything each session.

What actually drives output quality

🎯
Specificity of ask
Vague asks ("improve this") get generic output. Specific asks ("rewrite the approval card copy using tone-guide rules") get precise output. More tokens in → better output.
📄
Context in window
When I've read the relevant doc this session, quality goes up. When I'm working from memory (old session), I make errors. CLAUDE.md partly solves this.
🔁
Iteration rounds
Each feedback round costs tokens but produces better output. 3 tight rounds beats 1 open-ended ask. Lock decisions before asking me to build.
📏
Output length
Long outputs cost more. If you want a summary, say "brief" or "one paragraph." If you want depth, say "comprehensive." I default to matching the complexity of your ask.
📂
How many files I read
Reading 5 docs = ~30k tokens before I've said a word. Always worth asking: "do you need to read X or do you already know it?" if it's in CLAUDE.md.
🧠
Session freshness
New session = fresh 200k window, but I've lost all previous context. Good CLAUDE.md + STATUS.md means I recover fast without re-reading everything.

How to get more out of fewer tokens

✓ Do this
Start with "read CLAUDE.md first" — one read recovers all context cheaply.

Lock decisions before building. Changing direction mid-prototype costs 3× more than deciding upfront.

Batch related tasks in one message: "read app-spec, then write the error-states doc" — one read, one output.

Tell me the output format upfront. "Give me a 3-bullet summary" vs. "explain thoroughly" are very different token costs.
✗ Avoid this
Open-ended "review everything" without a focus area — I'll read all docs and produce a huge output.

Iterating on something before the spec is settled. Every rebuild round is 40–80k tokens.

Asking me the same question across sessions without updating CLAUDE.md — I'll re-derive the same answer from scratch.

Long back-and-forth chat before the actual task — conversation history piles up fast.

Recommended session structure (for this project)

1
Open with: "Read CLAUDE.md and STATUS.md"
~2k tokens. Gets me fully up to speed without touching the larger docs. I'll flag if I need more.
2
State the specific goal for this session
e.g. "Today we're writing error-empty-states.md" — I'll read only what's needed for that task, not everything.
3
Decide before I build
Ask clarifying questions first (cheap). Building the wrong thing and iterating is 5× more expensive than a 3-message scoping conversation.
4
End with: "Update CLAUDE.md and STATUS.md"
~1–2k tokens. Locks in decisions so the next session doesn't pay to rediscover them. This is the highest ROI action in the whole workflow.

What this guide can't do
This is estimated, not live. In Cowork mode, the actual token counts from the API aren't surfaced to me — I can't read them in real time. For exact counts, you'd need to call the Anthropic API directly and read the usage field in the response (input_tokens, output_tokens). The figures here are based on typical patterns for this kind of work, not live measurements.
