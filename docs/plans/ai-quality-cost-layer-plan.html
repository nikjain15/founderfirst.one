1. Why we're building this

Penny uses AI in ~11 separate places (website chat, Discord, insights, email drafts, signals scoring, …) across three runtimes (Cloudflare Workers, Supabase Edge Functions, a local Mac host). Each was built independently — its own model choice, its own error handling, no shared records.

That creates four problems that get worse as we grow and add more products:

  • Cost balloons. Expensive models used even for easy tasks; no shared caching. Cost per customer rises with volume instead of falling.
  • Quality is invisible. No record of which model answered, how good it was, or whether a human had to fix it. We can't prove quality is going up.
  • Stakes are high. This is bookkeeping — a wrong number or wrong tax claim is a real liability, not a demo bug.
  • It doesn't reuse. Every future agent re-solves the same hard problem from scratch.
The real scaling law for an autonomous product: the binding constraint isn't tokens — it's human-review attention. We only scale if the share of decisions resolved without a human climbs over time, safely. Cost is downstream of that.

A gift we found in the code: we already record fragments of this — model on insight_runs, confidence on insight_actions, intent/model on sig_scores, outcome on sig_leads, bot_confidence on tickets — in five incompatible shapes. This layer unifies them into one source of truth. It's a tidy-up, not a green field.

2. What it is (plain English)

One "middle desk" every AI request goes through. A product declares what it needs; the desk decides how to run it and keeps getting better on its own. It does six jobs:

  1. Routes to the cheapest model that's good enough (cheap for easy, frontier only when hard).
  2. Records every request — model, confidence, cost, speed, outcome.
  3. Audits — a second AI grades each answer against that use case's evals; risky ones are blocked or sent to a human.
  4. Queues for human review when needed.
  5. Reuses & learns — caches answers; repeated approved answers graduate into rules (no model call).
  6. Shows it — cost + quality trends and controls on the FounderFirst admin.

4. The evals (quality checks)

Ordered by criticality — most harmful if wrong, first. gate = blocks or escalates; the answer never reaches the user if it fails. score = tracked to drive improvement, not blocking. Per D7, each use case picks and tunes its own set above the mandatory floor (D8).

Tracked alongside (not answer-quality): cost per task and speed — the "cost going down" half of the goal.

6. Improvement loop & autonomy ramp

Every block, edit, rejection, and later correction is a lesson. Patterns drive fixes (better prompt, a new rule, a model swap). Two specific mechanisms:

  • Rule graduation. A repeated, human-approved answer (e.g. "vendor AWS → Software", 50×) becomes a deterministic rule that fires with no model call — cheaper and more consistent. Tenant-scoped (one customer's rule never fires on another's books).
  • Autonomy ramp (D5). 100% review → the system recommends reducing review when evals + approval stay high with zero safety/privacy misses → you approve → less review. Quality dips ⇒ snaps back automatically.
answer produced→ evals run→ gate: pass / block / escalate→ human verdict (early)→ recorded→ patterns → rules + ramp

8. The admin views

Three screens on the FounderFirst admin (reusing existing patterns — KPI strips, tables, review surfaces):

  • Overview — KPI strip (total spend, quality, autonomy %, awaiting review); per-use-case table with total cost, cost/task, quality, model in use, review mode, status; and a "what to do next" actions feed (apply cheaper route, approve more autonomy, see a failing pattern).
  • Use-case detail — cost breakdown + cost-by-model; editable models & routing (main / backup / hard-questions / spend cap / caching) with "test before saving"; quality checks; autonomy dial + recommendation; what's working; recent answers.
  • Eval setup (per use case) — table of applied evals (method, gate/score, threshold, live pass-rate), add from library or create custom, locked mandatory gates, "save to this use case" vs "save to library", "test on recent answers".

10. Guardrails document v1 (living)

The written rulebook the system follows. Starts here, updated as we learn; later editable in admin.

Mandatory floor (cannot be disabled)

  • Customer-facing use cases must run the Safe and Privacy gates at all times.
  • Financial outputs (numbers, categories) must run the Truthful/grounded and Math-adds-up gates.
  • No output ships if a gate fails — it is blocked or escalated to a human.

Autonomy-ramp rules (placeholder thresholds — tune with real data)

  • A use case may move from 100% review → sampling only after: ≥ [N] decisions, ≥ [X]% human-approval, and zero safety/privacy failures over [T] weeks.
  • Each reduction is proposed by the system and approved by a human (D4). Approval is audit-logged.
  • Rollback: if any gate-failure or approval-rate drops below [Y]% over [rolling window], the use case auto-reverts to the previous (stricter) review level.

Model-change rules

  • Model/routing changes must pass "test on recent answers" before going live.
  • Spend caps per use case are enforced by the gateway; hitting a cap triggers fallback, not failure.

11. Assumptions & risks to stress-test

The deliberate list of things we should attack before building, so we don't miss anything.

Assumptions baked in (validate each):
  • That an AI judge is reliable enough to gate customer-facing answers without too many false blocks (or missed bad answers).
  • That adding a judge + evals to live chat won't add noticeable latency.
  • That a per-use-case eval config in Supabase is expressive enough for real bookkeeping checks (not just text grading).
  • That Cloudflare AI Gateway covers our routing needs and that dropping local Ollama from the runtime path is acceptable.
  • That "both at once" (D1) is realistic given bookkeeping isn't live — i.e. we won't over-design for a product whose shape may change.
  • That rule graduation can be made safe in a multi-tenant financial system (no cross-customer leakage).
Top risks:
  • Judge cost & latency: grading every answer doubles model calls — could erode the cost savings or slow chat.
  • False confidence: evals that pass but miss real errors → we ramp autonomy on a bad signal.
  • Scope creep: building for bookkeeping before it exists.
  • Source-of-truth drift: if Langfuse (or any second store) creeps in, we re-create the very fragmentation we're fixing.
  • Migration safety: wiring 11 call sites and retiring scattered columns without breaking live surfaces.

12. Things we deliberately avoid

  • Always-on infra we babysit: LiteLLM, Kong, self-hosted Langfuse/Helicone/Argilla, GPTCache, self-hosted Redis.
  • Dead/fading tools: OpenAI Evals (shutting down), Humanloop (shut down), Helicone (maintenance mode).
  • A second source of truth for records/queue/dashboard outside Supabase + admin (D9).
  • Hardcoded model choices per file (D10).
  • Mac-host dependencies in any customer-facing path.

Freshness caveat: a few "fading/shutting down" facts came from secondary 2026 write-ups — confirm first-party before betting the roadmap on any one of them.
