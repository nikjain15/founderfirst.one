Penny — AI Quality & Cost Layer

A single layer every AI request passes through: routes to the right model, records everything, audits quality before it reaches users, and earns more autonomy as it proves itself — built on the existing Cloudflare + Supabase stack.

1. Why we're building this

Penny uses AI in ~11 separate places (website chat, Discord, insights, email drafts, signals scoring, …) across three runtimes (Cloudflare Workers, Supabase Edge Functions, a local Mac host). Each was built independently — its own model choice, its own error handling, no shared records.

That creates four problems that get worse as we grow and add more products:

  • Cost balloons. Expensive models used even for easy tasks; no shared caching. Cost per customer rises with volume instead of falling.
  • Quality is invisible. No record of which model answered, how good it was, or whether a human had to fix it. We can't prove quality is going up.
  • Stakes are high. This is bookkeeping — a wrong number or wrong tax claim is a real liability, not a demo bug.
  • It doesn't reuse. Every future agent re-solves the same hard problem from scratch.
The real scaling law for an autonomous product: the binding constraint isn't tokens — it's human-review attention. We only scale if the share of decisions resolved without a human climbs over time, safely. Cost is downstream of that.

A gift we found in the code: we already record fragments of this — model on insight_runs, confidence on insight_actions, intent/model on sig_scores, outcome on sig_leads, bot_confidence on tickets — in five incompatible shapes. This layer unifies them into one source of truth. It's a tidy-up, not a green field.

2. What it is (plain English)

One "middle desk" every AI request goes through. A product declares what it needs; the desk decides how to run it and keeps getting better on its own. It does six jobs:

  1. Routes to the cheapest model that's good enough (cheap for easy, frontier only when hard).
  2. Records every request — model, confidence, cost, speed, outcome — under one tenant-scoped, crash-safe log.
  3. Audits — deterministic checks first, then a panel of AI checkers using different model families grades each answer; agreed answers pass, only failures/disagreements go to a human (early on, a human also samples the passes).
  4. Queues for human review when needed.
  5. Reuses & learns — caches answers; repeated approved answers graduate into rules that are still re-checked through the gates.
  6. Shows it — cost + quality trends and live controls on the FounderFirst admin.

D15 newTenant isolation is a deterministic data-layer invariant. tenant_id NOT NULL on ai_decisions, the cache, and the rule table; a required tenant predicate on every query, asserted by a CI test. The Privacy eval is defense-in-depth, never the boundary.The stack runs as SERVICE_ROLE (RLS bypassed) with no tenant_id today; a missing predicate is a silent cross-tenant leak an AI eval can't catch.
D16 newGrounding is two checks: (a) source-exists (cheap rule) + (b) source-is-correct — a deterministic SQL reconciliation that the cited transaction/period/account matches the claim's tenant, date window, and account. Financial use cases cannot advance on the ramp until (b) exists as code.The dominant bookkeeping error is a real number on the wrong invoice/period — it passes a "number exists" check cleanly.
D17 newRule graduation never bypasses the gates. Rule/cache-fired decisions are periodically re-run through the math/grounding gates + a sampled judge; each graduated rule carries an expiry + re-validation and auto-demotes on CPA correction; rules/cache are tenant-partitioned by primary key and learned mappings are never promoted to a shared scope.A no-model-call rule otherwise skips every gate, laundering a one-time error into permanent, invisible autonomy.
D18 newLogging is async & crash-safe. The decision-log write fires via ctx.waitUntil() after the response returns, never blocking the reply; it log-drops with a counter if Supabase is unavailable (the answer still ships). Eval results are a JSONB column on the single record.A synchronous write would add latency and let a Supabase hiccup take Penny chat down.
D19 amendedRetention + erasure are first-class. Raw input/answer is kept 90 days, then archived (de-identified) to train our own models (D24) — not silently purged. A right-to-erasure request cascades to ai_decisions + judge rationale + cache/Vectorize and the archive, per the law. Personal details are stripped before data enters any training set, so a trained model is never tied to an individual.Serves the training goal while keeping legal deletion achievable (you can't un-train a model). Ties to the open Discord erasure obligation.
D20 newJudge robustness. Customer input is treated as data, never instructions (structured delimiting + instruction hierarchy); the locked floor has a deterministic non-LLM check; injection canaries live in the eval suite; the judge is a different model family than the generator on gate evals, and a stronger model guards the locked financial gates.A vendor memo can prompt-inject the judge to PASS; a same-model judge shares the generator's blind spots.
D21 newMigration safety. resolve() dual-writes the legacy column + ai_decisions; a scheduled job reconciles them on a sample for N days; consumers flip to read ai_decisions one surface at a time behind a flag; legacy columns are retired last. Migrations applied manually.Live admin surfaces read those scattered columns today; cutting writes silently shows stale data (LEARNINGS rule 14, rule 3).
D22 newHonest cost accounting. "Judge cost as % of answer cost" and "cost per resolved task" (incl. retries/escalations + human time) are first-class KPIs with caps. The $0–29/mo figure is infra-only; per-answer cost is net-neutral-to-up until caching + rule graduation mature.The judge adds 2–3× tokens where volume is highest; the savings mechanisms come later in the build — say so plainly.
D23 newNo scope-cut. This is a fully-functioning, well-structured product. All six phases are committed build milestones; nothing is shipped as a thin pilot or left optional.Per directive — robustness over speed-to-a-demo.
D24 newTrain our own models from the collected data to cut dependence on expensive external models over time. Training uses de-identified data only; the model-training pipeline is itself a use case that flows through the same gates and records.Compounding margin + a moat: more usage → better owned models → lower cost. The de-identify step is what keeps legal deletion possible.
D25 newValidate the panel, don't just trust it. Early in each use case, a human reviews a small random sample of the answers the panel PASSED — not only the failures — to confirm the panel is actually right. The sample shrinks as the panel proves out (this is the autonomy ramp, D5).A panel can be confidently wrong together; sampling the passes is how we earn the right to trust it.

6. Improvement loop & autonomy ramp

Every block, edit, rejection, and later correction is a lesson. Two mechanisms:

  • Rule graduation (D17). A repeated, human-approved answer becomes a fast rule — but it is still periodically re-run through the math/grounding gates + a sampled judge, carries an expiry, auto-demotes on CPA correction, and is tenant-partitioned (never shared across customers).
  • Autonomy ramp (D5). 100% review → the system recommends reducing review when zero-edit approval + evals stay high with zero safety/privacy misses (per tenant cohort) → you approve → less review, with a continuous shadow sample. Rollback triggers on the lagging CPA/customer-correction rate.
answer produced→ deterministic gates→ panel of diverse models→ agree → pass · disagree/fail → human→ human verdict (zero-edit?) + sampled passes→ async record→ patterns → gated rules + ramp

8. The admin views

Three screens on the FounderFirst admin (reusing existing patterns):

  • Overview — KPI strip (total spend, quality, autonomy %, awaiting review, judge cost as % of answer cost, cost per resolved task); per-use-case table with total cost, cost/task, quality, model in use, review mode, status; and a "what to do next" actions feed.
  • Use-case detail — cost breakdown + cost-by-model; editable models & routing (main / backup / hard-questions / spend cap / caching) with "test before saving"; quality checks; autonomy dial + recommendation; what's working; recent answers (showing gate outcome incl. fail-closed).
  • Eval setup (per use case) — applied evals (method, gate/score, threshold, live pass-rate), add from library or create custom, locked mandatory gates, save-to-use-case vs save-to-library, "test on recent answers".

10. Guardrails document v2 (living)

Mandatory floor (cannot be disabled)

  • Customer-facing use cases run the Safe and Privacy/tenant-isolation gates at all times, enforced by deterministic non-LLM checks (D8, D15, D20).
  • Financial outputs run Source-exists, Source-correct, and Math gates (D16).
  • No output ships if a gate fails. On judge timeout/error, fail CLOSED to a templated human-handoff (D3).
  • Tenant isolation: tenant_id NOT NULL + required query predicate everywhere; the Privacy eval is defense-in-depth, never the boundary (D15).

Autonomy-ramp rules (placeholder thresholds — tuned with real data + reviewer capacity)

  • Advance from 100% review → sampling only after: ≥ [N] decisions, ≥ [X]% zero-edit approval, and zero safety/privacy failures over [T] weeks, per tenant cohort.
  • Each reduction is proposed by the system and approved by a human, audit-logged (D4).
  • Rollback on the lagging CPA/customer-correction rate falling below [Y]% (rolling window); a shadow review sample runs continuously post-ramp.
  • Financial use cases cannot advance until the Source-correct reconciliation gate (D16) exists as code.
  • Graduated rules are re-validated through the gates, expire, and auto-demote on correction (D17).

Model, judge & cost rules

  • Gate-eval judge is a different model family than the generator; locked financial gates use a stronger judge (D20).
  • Model/routing changes pass "test on recent answers" before going live (D10).
  • Spend caps per use case are enforced (gateway or resolve()); hitting a cap triggers fallback, not failure (D11).
  • "Judge cost as % of answer cost" stays under [cap]; score-evals are sampled, not run at 100% (D12, D22).

Data rules

  • Raw input/answer purged after the review+ramp window; scores + hashed refs retained for trends (D19).
  • Right-to-erasure cascades to ai_decisions + judge rationale + cache/Vectorize; immutable audit fields are separated from purgeable PII (D19).
  • Gateway body-logging minimized for financial use cases; gateway/exact-match cache keyed by tenant or off for financial (D11).

12. Things we deliberately avoid

  • Always-on infra we babysit: LiteLLM, Kong, self-hosted Langfuse/Helicone/Argilla, GPTCache, self-hosted Redis.
  • Dead/fading tools: OpenAI Evals (shutting down), Humanloop (shut down), Helicone (maintenance mode).
  • A second source of truth for records/queue/dashboard outside Supabase + admin (D9).
  • Hardcoded model choices per file (D10).
  • Mac-host dependencies in any customer-facing path.
  • An AI eval as a security boundary — isolation and the locked floor are deterministic code (D15, D20).
  • Rules/cache that skip the gates or cross tenants (D17).
